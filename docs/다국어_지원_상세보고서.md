## 1. 문제의 정확한 정의 — 두 개의 서로 다른 레이어

다국어 지원은 하나의 작업이 아니라 **완전히 독립된 두 개의 레이어**로 이루어진다. 이 둘을 섞어서 생각하면 안 된다.

```
레이어 1: 스토어 페이지 (Play Console)     레이어 2: 앱 내부 UI (Flutter 코드)
─────────────────────────────           ─────────────────────────────
사용자가 스토어에서 앱을 "발견"할 때        사용자가 앱을 "설치해서 실행"할 때
보이는 화면                                보이는 화면

이미 표준 기능으로 지원됨                   현재 미구현 (한국어 하드코딩)
작업량: 낮음 (텍스트+이미지 번역만)          작업량: 높음 (전체 코드 리팩토링)
```

### 왜 이 구분이 중요한가
"영어판이 있다"는 말이 두 가지 의미로 오해될 수 있다:
- (A) "영어권 사용자가 스토어에서 검색하면 영어로 된 소개를 본다" → 레이어 1만 있어도 충족
- (B) "영어권 사용자가 앱을 설치해서 실행하면 앱 자체가 영어로 나온다" → 레이어 2까지 있어야 충족

지금까지 (A)와 (B)를 구분하지 않고 "다국어"라고 뭉뚱그려 얘기했기 때문에, **정확히 무엇이 준비됐고 무엇이 안 됐는지 착오가 생길 수 있는 지점**이다.

## 2. 레이어 1 — 스토어 페이지 다국어 (Play Console)

### 2-1. 작동 원리
Play Console은 앱 하나에 대해 **언어별로 완전히 다른 스토어 등록정보 세트**를 저장할 수 있다.

```
기본값(한국어) 등록정보
├── 앱 이름: PrayStory - 나의 서신서
├── 짧은 설명: (한국어)
├── 자세한 설명: (한국어)
├── 스크린샷: (한국어 UI 캡처본)
└── 피처 그래픽: (한국어 텍스트 포함 이미지)

영어(en-US) 등록정보 [별도 추가 가능]
├── 앱 이름: PrayStory
├── 짧은 설명: (영어)
├── 자세한 설명: (영어)
├── 스크린샷: (영어 UI 캡처본 — 앱이 영어 지원 안 하면 한국어 UI 그대로 써도 무방)
└── 피처 그래픽: (영어 텍스트 포함 이미지, 또는 한국어와 동일 이미지 재사용 가능)
```

**노출 로직**: 사용자의 Google Play 계정/기기 언어 설정을 기준으로 자동 분기. 별도 사용자 조작 불필요.

### 2-2. 설정 위치
```
Play Console → 해당 앱 → 스토어 등록정보 → 기본 스토어 등록정보
→ 상단 "번역 관리" 버튼 클릭
→ "언어 추가" → English (United States) 선택
→ 각 필드 영어로 입력
```

### 2-3. 필요한 작업물 (번역만 하면 됨, 코드 불필요)

| 항목 | 현재 상태 | 필요 작업 |
|---|---|---|
| 앱 이름 | "PrayStory - 나의 서신서" | 영어권엔 "PrayStory"만 노출해도 무방 |
| 짧은 설명 (80자) | 한국어 확정됨 | 영어 번역 필요 |
| 자세한 설명 (4000자) | 한국어 확정됨 (본 보고서 하위 참고) | 영어 번역 필요 |
| 스크린샷 5장 | 한국어 UI로 캡처됨 | **레이어 2가 없으면 한국어 UI 그대로 사용** (영어권 사용자도 이해 가능한 수준으로, 또는 캡션만 영어로 오버레이) |
| 피처 그래픽 | 검정 버전 완성 | 텍스트("나의 서신서")를 영어("PrayStory")로 교체한 버전 별도 제작 필요 |

## 3. 레이어 2 — 앱 내부 UI 다국어 (실제 코드 작업)

### 3-1. 현재 상태 진단

프로젝트에 이미 다음이 세팅되어 있음:
```dart
// main.dart
localizationsDelegates: const [
  GlobalMaterialLocalizations.delegate,
  GlobalWidgetsLocalizations.delegate,
  GlobalCupertinoLocalizations.delegate,
],
supportedLocales: const [
  Locale('ko', 'KR'),
  Locale('en', 'US'),
],
```

**이건 "다국어를 지원할 준비"이지 "다국어 지원 자체"가 아니다.** 이 설정은 Flutter의 기본 위젯(날짜 피커, 스크롤바 등)이 시스템 언어에 맞게 나오도록 하는 것뿐이고, **앱이 직접 그린 텍스트("나의 서신서", "오늘 하루...", 버튼 문구 등)는 전부 한국어로 하드코딩되어 있어서 이 설정과 무관하게 항상 한국어로 뜬다.**

예시 (실제 코드에서 확인된 하드코딩 패턴):
```dart
Text('나의 서신서', style: ...)
hintText: '기도 제목',
'성경책 열기', '서신서 시작하기',  // 버튼 문구
'오늘 하루, 당신의 삶에\n행하신 하나님의 이야기를\n기록해 보세요.',  // 안내 문구
```

이런 패턴이 코드 전체에 수십~백여 곳 흩어져 있을 것으로 추정됨 (정확한 개수는 실제 코드 검색 필요).

### 3-2. 목표 아키텍처

Flutter 표준 방식인 **`flutter_localizations` + `.arb` 파일** 기반으로 전환:

```
lib/
├── l10n/
│   ├── app_ko.arb   ← 한국어 원본 문자열
│   └── app_en.arb   ← 영어 번역 문자열
└── main.dart         ← AppLocalizations 델리게이트 추가
```

사용 코드는 이렇게 바뀜:
```dart
// 기존 (하드코딩)
Text('나의 서신서')

// 변경 후 (리소스 참조)
Text(AppLocalizations.of(context)!.appTitle)
```

### 3-3. 필요한 패키지

`pubspec.yaml`에 이미 있는 `flutter_localizations` 외에 추가 필요:
```yaml
dependencies:
  flutter_localizations:
    sdk: flutter
  intl: ^0.20.2  # 이미 있음

flutter:
  generate: true  # .arb 파일 자동 코드 생성 활성화
```

`l10n.yaml` 설정 파일 신규 생성 필요:
```yaml
arb-dir: lib/l10n
template-arb-file: app_ko.arb
output-localization-file: app_localizations.dart
```

### 3-4. 작업 범위 추정 — 화면별 문자열 개수 (추정치)

실제 코드를 정확히 검색해야 확정 가능하지만, 지금까지 파악된 화면 구조 기준으로 추정:

| 화면 | 예상 하드코딩 문자열 수 | 비고 |
|---|---|---|
| 인증(로그인/회원가입) | ~15개 | 버튼, 힌트텍스트, 에러메시지 |
| 홈/서신서 메인 | ~10개 | 안내문구, 날짜 포맷 |
| 기도 작성/수정 | ~10개 | 힌트, 버튼, 확인 다이얼로그 |
| 기도 기록(달력/통계) | ~15개 | 통계 라벨, 달력 요일 |
| 커뮤니티(그룹) 화면 7종 | ~40개 | 화면 수가 많아 문자열도 많음 |
| 설정 화면 | ~10개 | |
| 알림 관련 | ~10개 | 알림 제목/본문 템플릿 |
| 공통(에러메시지, 스낵바 등) | ~15개 | |
| **합계 (추정)** | **약 120~140개** | |

이 정도 규모면 **하루~하루 반나절 정도의 집중 작업**으로 예상됨 (기계적 치환 작업이 대부분이라 시간 자체는 오래 안 걸리나, 빠짐없이 다 찾아서 바꾸는 꼼꼼함이 필요).

### 3-5. 특히 주의해야 할 부분 — 단순 텍스트가 아닌 것들

일반 `Text('문구')` 치환보다 까다로운 부분들:

**① 날짜/요일 포맷**
```dart
DateFormat('yyyy년 M월 d일 (E)', 'ko').format(date)
```
한국어는 "2026년 6월 24일 (수)", 영어는 "June 24, 2026 (Wed)" 형식이 완전히 다름. `intl` 패키지의 로케일별 포맷 사용 필요, 단순 문자열 치환 불가.

**② 복수형 처리**
```dart
'$count일 연속 기록'  // 한국어는 단복수 구분 없음
```
영어는 "1 day streak" vs "3 days streak"처럼 단수/복수가 다름. `.arb` 파일의 `plural` 문법 사용 필요.

**③ 알림(Push Notification) 텍스트**
`notification_service.dart`의 알림 제목/본문도 하드코딩되어 있음:
```dart
'기도 시간이에요', '오늘을 위한 기도 제목을 확인해 보세요.'
```
이건 앱이 백그라운드에 있을 때도 발송되므로, 알림 발송 시점에 사용자의 언어 설정을 읽어와서 분기하는 로직 필요.

**④ 익명 닉네임 생성 (커뮤니티)**
```dart
_anonymousAdjectives = ['gentle', 'quiet', 'warm', ...]  // 이미 영어
_anonymousNouns = ['lamb', 'dove', 'sparrow', ...]      // 이미 영어
```
이건 이미 영어로 되어 있어서 오히려 손댈 필요 없음 (다만 한국어 사용자에게 "gentle-lamb" 같은 닉네임이 그대로 노출되는 게 의도인지는 확인 필요 — 원래 설계 그대로 유지해도 무방해 보임).

**⑤ 성별 선택지("남"/"여")**
```dart
['남', '여'].map(...)
```
영어는 "Male"/"Female" 등으로 바뀌어야 하고, DB에 저장되는 값(`gender` 필드)을 한국어로 저장할지 코드값(M/F)으로 저장할지도 재검토 필요 — 지금은 한국어 문자열 그대로 DB에 저장되는 구조라, 다국어 전환 시 **기존 사용자 데이터와의 호환성**도 고려해야 함.

### 3-6. 언어 전환 방식 — 두 가지 옵션

**옵션 A: 기기 언어 자동 감지만 지원 (권장, 간단)**
- 사용자가 뭘 선택할 필요 없이, 폰 시스템 언어가 영어면 앱도 자동으로 영어
- 구현: `supportedLocales`에 이미 등록된 로케일 중 기기 설정과 매칭되는 것을 Flutter가 자동 선택 (별도 로직 거의 불필요)

**옵션 B: 앱 내 언어 선택 기능 추가 (더 유연하지만 작업량 증가)**
- 설정 화면에 "언어" 메뉴 추가, 기기 언어와 무관하게 사용자가 직접 선택
- 구현 필요 사항:
  - 선택한 언어를 `SharedPreferences`에 저장
  - `MaterialApp`의 `locale` 파라미터를 저장된 값으로 오버라이드
  - 설정 화면에 언어 선택 UI(라디오 버튼 또는 드롭다운) 추가

**권장**: 1차로는 **옵션 A만 구현**하고, 사용자 요청이 실제로 있으면 옵션 B를 나중에 추가하는 것이 합리적 (개발 리소스 대비 가치 판단).

### 3-7. 작업 순서 제안 (Claude Code 진행 시)

```
1단계: .arb 파일 인프라 구축
  - l10n.yaml 생성
  - lib/l10n/app_ko.arb, app_en.arb 빈 틀 생성
  - pubspec.yaml에 generate: true 추가
  - main.dart에 AppLocalizations 델리게이트 연결
  - flutter gen-l10n 실행해서 코드 생성 확인 (에러 없이 빌드되는지)

2단계: 화면 단위로 순차 치환 (한 화면씩, 검증하며 진행)
  - 인증 화면부터 시작 (진입점이라 우선순위 높음)
  - 홈/서신서 메인
  - 기도 작성/수정
  - 기도 기록
  - 커뮤니티 (화면 수 많아 가장 오래 걸림)
  - 설정
  - 공통 컴포넌트(다이얼로그, 스낵바 등)

3단계: 특수 케이스 처리
  - 날짜 포맷 로케일 분기
  - 복수형(streak 일수 등)
  - 알림 텍스트 다국어화
  - 성별 등 DB 저장값 재검토

4단계: 영어 번역 검수
  - 초벌 번역은 AI로 가능하나, 신앙 용어 뉘앙스는 재검토 권장
  - 실기기에서 시스템 언어를 영어로 바꿔서 전체 화면 순회 테스트

5단계: 회귀 테스트
  - 한국어 모드에서 기존 동작 그대로인지 확인 (문자열 리소스화 과정에서 오타/누락 없는지)
```

### 3-8. 데이터베이스 관련 — 놓치기 쉬운 지점

Supabase에 저장된 콘텐츠 자체(사용자가 쓴 기도문 등)는 번역 대상이 아님(사용자 생성 콘텐츠는 그 언어 그대로 유지되는 게 당연). 다만:

- `profiles` 테이블의 `gender` 필드가 지금 한국어 문자열("남"/"여")로 저장되고 있어서, 영어 UI에서 이 값을 어떻게 표시할지 매핑 로직 필요 (예: DB엔 "남" 그대로 두고, 화면 표시할 때만 `gender == '남' ? 'Male' : 'Female'`로 변환)
- 알림 채널 이름("기도 알림", "내일을 위한 기도") 등 `notification_service.dart`의 문자열도 시스템 설정 채널명이라 사용자에게 노출되므로 다국어 고려 대상

## 4. 두 레이어의 우선순위 — 실행 판단

| 시나리오 | 필요한 레이어 | 예상 공수 |
|---|---|---|
| **1차 배포 최소 버전** | 레이어 1만 (스토어 페이지 영어 번역) | 번역 작업 1~2시간 |
| **완전한 글로벌 대응** | 레이어 1 + 레이어 2 | 레이어 2 추가로 1~2일 |

### 이전 대화에서 제안했던 절충안 (재확인)
> "1차 배포는 한국어 앱 + 영어 스토어 페이지만으로 가는 절충안" — 이건 실제로 많은 로컬 앱들이 택하는 현실적인 방식. 스토어에서 영어로 사람을 데려온 뒤, 앱을 열었을 때 한국어가 나와도 이미 많은 다국어 미지원 로컬 앱들이 이렇게 시작함. 다만 이 경우 스토어 설명에 "현재 한국어만 지원됩니다(Currently Korean only)" 같은 문구를 명시하는 것이 사용자 이탈/리뷰 불만을 줄이는 데 도움이 됨.

## 5. 결정이 필요한 사항 (아직 미정, 사용자 판단 필요)

1. **레이어 2(앱 내부 다국어)를 1차 배포에 포함할지, 이후 업데이트로 미룰지**
2. 포함한다면 **옵션 A(자동감지)만 할지, 옵션 B(수동 선택)까지 할지**
3. 영어 번역 초안(위 3-6 등)을 **원어민 검수 없이 그대로 쓸지, 검수를 거칠지**
4. `gender` 필드처럼 **기존 데이터 구조를 다국어 대응을 위해 변경할지** (스키마 변경은 신중해야 함 — 이미 배포된 앱이라면 마이그레이션 이슈 고려)
