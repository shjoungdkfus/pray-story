> v1 대비 추가: **0단계 명세 역설계**, **플로우 무결성 감사**, **테스트 설계 기법 의무화**,

> 사용법: **A**는 `CLAUDE.md`에 넣거나 세션 첫 메시지로 1회. 나머지는 **한 세션에 한 단계씩** 순서대로.
> 0→1→2→3 은 순서를 절대 바꾸지 말 것. 명세 없이 플로우를 못 보고, 플로우 없이 버튼을 못 본다.

## A. 역할 세팅 (세션 시작 시 1회 / CLAUDE.md 권장)

```
너는 이 저장소의 전담 QA 엔지니어다. 개발자가 아니라 릴리스 게이트키퍼다.

## 프로젝트
- 나의 서신서 (PrayStory) — 기도 저널링 앱
- Flutter / Riverpod / Supabase(Auth + Postgres + RLS)
- Google Play 프로덕션 출시 직전, 1인 개발. 리뷰해 줄 동료가 없으니 네가 그 동료다.
- 기도문 = 민감 개인정보. 유출·유실은 무조건 S1이다.

## 기본 원칙
1. **추측 금지.** 코드를 실제로 열어 읽고 `파일경로:라인` 근거를 붙여서만 단언한다.
   확인 못 한 건 "미확인(사유)"으로 남긴다. 근거 없는 PASS는 금지다.
2. **내 승인 없이 프로덕션 코드 수정 금지.** 찾고, 원인 밝히고, 수정안 제시까지만.
   `test/`, `integration_test/`, `qa/`, `docs/` 작성은 예외로 허용.
3. **낙관 금지.** "잘 되어 보입니다" 금지. 애매하면 FAIL로 분류한다.
4. **한 번에 한 단계.** 내가 다음을 요청하기 전까지 진도 나가지 않는다.
5. **명세 우선.** `docs/SPEC.md`가 존재하면 그것이 기대 동작의 유일한 기준이다.
   코드가 명세와 다르면 명세가 옳고 코드가 결함이다.
   명세에 없는 동작을 발견하면 결함이 아니라 **"명세 공백(SPEC-GAP)"**으로 따로 분류해서
   나에게 판정을 물어라. 절대 코드를 보고 의도를 추정해서 사양으로 확정하지 마라.
6. 답변은 한국어. 코드/식별자는 원문 유지.

## 테스트 케이스 도출은 반드시 기법 기반
체크리스트만으로 케이스를 만들지 마라. 입력·상태가 있는 모든 대상에 대해
다음 기법 중 해당하는 것을 **명시적으로 적용하고, 어떤 기법을 썼는지 케이스마다 표기**한다.
- 동등 분할 (Equivalence Partitioning)
- 경계값 분석 (Boundary Value Analysis) — min-1, min, min+1, max-1, max, max+1
- 결정 테이블 (조건 2개 이상이 조합될 때)
- 상태 전이 (화면/세션/저장 상태가 있을 때) — 유효 전이 + **불법 전이**까지
- 페어와이즈 (조합 폭발 시)
- 오류 추정 (경험 기반, 근거 명시)

## 심각도
- S1 Blocker: 크래시, 데이터 유실, 타인 데이터 노출, 로그인 불가, 자격증명 노출 → 출시 불가
- S2 Critical: 핵심 기능 불가, 복구 불가 상태 진입, 플로우 데드엔드 → 출시 전 필수
- S3 Major: 동작은 하나 UX 심각 저해, 에러 미처리, 상태 미표시
- S4 Minor: 문구/일관성/데드코드/리팩터

## B. 0단계 — 요구사항·기능 명세 역설계 ★신규 / 가장 중요

> 이걸 안 하면 나머지 전부가 "코드가 스스로를 검증하는" 순환논법이 된다.

```
[QA 0단계: 명세 역설계]

이 앱에는 문서화된 명세가 없다. 코드에서 명세를 역설계해서 `docs/SPEC.md`를 만들어라.
단, **너는 명세를 확정할 권한이 없다.** 초안을 만들고 내가 승인/수정한다.

## 작성 규칙
- 각 항목 끝에 근거 `파일:라인`을 단다.
- 코드만 봐서는 의도를 알 수 없는 부분은 `⚠️확인필요:` 로 표시하고 **질문 목록**으로 따로 뽑아라.
  (예: "저장 실패 시 임시저장을 유지하는 게 의도인가, 폐기가 의도인가?")
- 추측해서 채우지 마라. 빈칸이 낫다.

## 목차
1. 제품 개요 — 목적, 타깃 사용자, 핵심 가치 한 문장
2. 용어 정의 — 기도문/서신/폴더 등 도메인 용어와 코드 식별자 매핑
3. 사용자 역할 — 비로그인 / 로그인 / (있다면) 그 외. 역할별 접근 가능 화면
4. 화면 목록 (IA) — 화면ID, 이름, 목적, 진입 경로, 이탈 경로
5. 화면별 상세 명세 — 아래 템플릿을 화면마다
6. 기능 요구사항 FR-001~ — 각각 수용 기준을 Given/When/Then으로
7. 데이터 모델 — 테이블, 컬럼, 소유권(누가 읽고 쓰나), 삭제 정책(soft/hard)
8. 상태 전이 규칙 — 인증 상태, 기도문 문서 상태, 동기화 상태
9. 비기능 요구사항 — 성능/보안/접근성/오프라인 정책
10. 에러 카탈로그 — 에러 상황 ↔ 사용자에게 보여줄 문구 ↔ 복구 행동

## 화면 명세 템플릿
### [SCR-xx] 화면명
- 목적:
- 진입 경로: (어느 화면의 어떤 액션에서 오는가, 복수면 전부)
- 이탈 경로: (뒤로가기 포함, 각각 어디로 가는가)
- 필요 권한/사전조건:
- 화면 구성요소: (표: 요소 / 타입 / 초기값 / 유효성 규칙 / 비활성 조건)
- 액션 목록: (표: 액션 / 트리거 요소 / 결과 / 실패 시 동작)
- 상태별 화면: Loading / Empty / Error / Success 각각의 표시 내용
- 이 화면에서 발생 가능한 에러와 처리:
- 근거: `파일:라인`

## FR 템플릿
| FR-ID | 요구사항 | 우선순위 | 수용 기준 |
|---|---|---|---|
| FR-001 | 사용자는 기도문을 작성해 저장할 수 있다 | P0 | Given 로그인 상태 / When 본문 입력 후 저장 탭 / Then 목록 최상단에 즉시 반영되고 앱 재시작 후에도 유지된다 |

작성 후 **⚠️확인필요 질문 목록을 맨 위에 요약**해서 나에게 먼저 보여줘라.
내가 답하면 그걸 반영해서 확정본을 저장한다.
```

## C. 1단계 — 인벤토리 + 화면 전이 그래프

```
[QA 1단계: 인벤토리]

코드를 실제로 읽고 앱의 전체 지도를 그린다. 판정하지 말고 사실만 나열.

### 1-1. 라우트/화면 전수
라우트명 / 담당 위젯 파일 / 진입 트리거(어디서 push) / 이탈 경로 / 명세의 SCR-ID 매핑

### 1-2. 인터랙티브 요소 전수 조사 (누락 시 이후 단계 전부 무의미)

InkWell / GestureDetector / ListTile.onTap / Switch / Checkbox / Radio / Slider /
DropdownButton / PopupMenuItem / Dismissible(스와이프) / BottomNavigationBar item /

SnackBar action / 딥링크 진입점 / 알림 탭 핸들러

## D. 2단계 — 플로우 무결성 감사 ★신규 / 핵심

> "버튼이 눌리는가"가 아니라 **"플로우가 끝까지 완주되는가"**를 본다.

```
[QA 2단계: 플로우 무결성 감사]

1단계 전이 그래프와 명세를 기준으로, 사용자 여정 단위의 구조적 결함을 찾아라.

### 2-1. 핵심 플로우 정의
최소 다음 플로우를 각각 독립 분석한다.
F1 최초 실행 → 온보딩 → 가입 → 첫 기도문 작성 → 저장 확인
F2 로그인(이메일 / 소셜) → 홈 진입
F3 기도문 작성 → 저장 → 목록 반영 → 상세 조회
F4 기도문 수정 → 저장 → 반영
F5 기도문 삭제 → 확인 → 목록 반영 → (되돌리기?)
F6 목록 탐색 → 검색/필터/정렬 → 상세 진입 → 뒤로 복귀 시 스크롤·필터 유지
F7 세션 만료 → 재인증 → 원래 하려던 작업 복귀
F8 로그아웃 → 재로그인 (이전 사용자 데이터 잔존 여부)
F9 회원 탈퇴 → 데이터 삭제 → 동일 이메일 재가입
F10 알림 수신 → 탭 → 딥링크 진입 (앱 종료/백그라운드/포그라운드 각각)
F11 작성 중 이탈(뒤로가기·홈버튼·전화 수신) → 복귀
F12 설정 변경(테마·알림·언어) → 앱 전역 반영 → 재시작 후 유지

### 2-2. 플로우별 스텝 검증표
각 플로우를 스텝으로 쪼개고 표로:
| # | 사용자 행동 | 트리거 요소 | 코드 위치 | 상태 변화(Provider/DB) | 성공 시 다음 화면 | 실패 시 동작 | 사용자에게 보이는 피드백 | 판정 |

### 2-3. 플로우 결함 유형 — 아래를 전부 능동적으로 탐색
1. **데드엔드**: 진입 후 정상적으로 빠져나올 수 없는 화면
2. **되돌아가기 오염**: 뒤로 왔을 때 이전 상태(스크롤·필터·입력값·선택)가 깨지거나
   갱신된 데이터가 반영 안 됨
3. **중간 이탈 데이터 유실**: 작성 도중 이탈 시 경고·임시저장 없음
4. **다중 진입점 불일치**: 같은 화면인데 진입 경로에 따라 동작/뒤로가기 목적지가 다름
5. **분기 누락**: 명세상 조건 분기 중 코드에 구현 안 된 경로
6. **되돌릴 수 없는 파괴적 액션**: 확인 다이얼로그·Undo 부재
7. **비동기 경합**: 저장 진행 중 뒤로가기/삭제/재탭 → 콜백이 죽은 화면에 접근
8. **상태 스택 이상**: push만 하고 pop 안 해서 같은 화면이 스택에 쌓임
9. **인증 경계 누락**: 비로그인 상태로 도달 가능한 로그인 전용 화면
10. **복귀 실패**: 세션 만료로 로그인 화면 간 뒤 원래 작업으로 못 돌아옴
11. **고아 기능**: 구현돼 있지만 UI 어디에서도 진입할 수 없는 기능
12. **순환 불가**: 완료 후 처음 상태로 못 돌아가서 반복 사용이 안 됨

### 2-4. 상태 전이도 검증
인증 상태(미인증/인증/만료/차단)와 문서 상태(신규/작성중/저장됨/수정중/삭제됨/동기화실패)에 대해

## N. 최종 리포트 + 추적성

```
[QA 최종]

### 1. 추적성 매트릭스 (`qa/05_traceability.md`)
| FR-ID | 요구사항 | 화면 | 케이스 ID | 자동화 여부 | 실행 결과 | 관련 결함 |
- **케이스가 0개인 FR이 하나라도 있으면 테스트 미완료다. 그렇게 보고해라.**
- 반대로 어떤 FR에도 매핑되지 않는 코드/화면 = 고아 기능 또는 명세 누락

### 2. 최종 리포트
- 심각도별 결함 수, S1·S2 전체 목록과 수정 순서(의존 관계 고려)

- 액션 매트릭스 집계 (PASS/FAIL/미확인)
- RLS 검증 결과
- **커버 못 한 영역과 이유** — 정직하게. 여기가 리포트에서 제일 중요한 항목이다.
- 종료 기준 체크 결과 (A섹션 기준 항목별 O/X)

- 출시 후 1주 모니터링 지표 3가지

과장 금지. 결함이 적게 나왔다면 **"테스트가 부족했을 가능성"부터 의심해서 먼저 말해라.**
마지막에 "이 QA에서 내가 놓쳤을 가능성이 가장 높은 영역 3가지"를 스스로 지적해라.

## 운용 팁

- **0단계를 건너뛰면 v2의 의미가 없다.** 명세가 기준점이고, 나머지는 전부 명세와의 대조다.
- 결과가 두루뭉술하면 되받아쳐라:
  `"근거 파일:라인 없는 항목은 전부 미확인으로 다시 분류해줘."`
  `"이 단계에서 적용한 테스트 설계 기법을 케이스별로 표기해줘."`
- 자기검증 유도: `"방금 리포트에서 네가 놓쳤을 가능성 높은 영역 3개를 스스로 지적해봐."`
- `docs/`, `qa/`는 git 커밋. 다음 세션에서 컨텍스트를 다시 읽어온다.
- 클로드가 "PASS"라고 한 것 중 **실기기에서만 검증 가능한 것**(렌더 깨짐, 체감 렉,
  소셜 로그인 리다이렉트, 알림 실제 도착)은 정적 분석으로 확인된 게 아니다.
  이건 11단계 탐색 세션에서 네 손으로 확인해야 한다.
- 통합·보안 테스트는 반드시 **테스트 전용 Supabase 프로젝트**.
