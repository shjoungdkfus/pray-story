## 매 업데이트마다 해야 하는 작업 (자동 아님)
- `pubspec.yaml` 버전 bump (`versionCode`는 이전 값보다 커야 재업로드 가능)
- `flutter build appbundle --release` 로 AAB 재빌드
- Play Console에 업로드 + 심사 대기 (통상 몇 시간~하루)
- 업데이트가 잦아지면 GitHub Actions + Fastlane 또는 Google Play Publishing API로 빌드/업로드 자동화 고려

- 스토어 등록정보(이름/설명/스크린샷) — 기능이 크게 안 바뀌면 그대로 둬도 됨
- `android/key.properties` + `upload-keystore.jks` (서명 키, git 미추적 유지)
- 개인정보처리방침 URL (GitHub Pages: `docs/privacy_policy.html`, `docs/privacy_policy_en.html`)

### 콘텐츠 등급 설문 (IARC)

- 예상 등급: 전체이용가

### 데이터 보안(Data safety) 양식

**수집 데이터**

| 유형 | 수집 | 필수/선택 | 목적 |
|---|---|---|---|
| 이름 | 예 | 필수 | 앱 기능, 계정 관리 |
| 이메일 주소 | 예 | 필수 | 앱 기능, 계정 관리 |
| 사용자 ID | 예 | 필수 | 앱 기능, 계정 관리 |
| 기타 개인정보 (교회, 출생연도, 성별) | 예 | 선택 | 앱 기능 |
| 사용자 제작 콘텐츠 (기도 기록, 커뮤니티 편지) | 예 | 필수 | 앱 기능 |

- 공유 여부: 제3자와 데이터 공유 **안 함** (Supabase는 데이터 처리자로 분류, Google/Kakao는 인증 목적만)
- 전송 중 암호화: 예 (Supabase HTTPS/TLS)
- 사용자 데이터 삭제 요청 가능: 예 (설정 > 계정 탈퇴 기능, `account_screen.dart`)

⚠️ 확인 필요: "소중한 이에게" 익명 편지가 특정 상대 대상 다이렉트 메시지 방식이면 Play의 "Messages" 카테고리로도 잡힐 수 있음. 전체/그룹 공개 피드 형태면 위 "사용자 제작 콘텐츠"로 충분. 실제 UI 동작 방식에 맞게 Play Console에서 최종 판단.

### Edge Function 배포 완료 + API 테스트로 검증 (2026-07-16 새벽)
- `npx supabase` CLI로 프로젝트 링크 후 `delete-account` 함수 배포 완료 (Docker 불필요, 그냥 됨)

  - **1차 테스트에서 버그 발견**: A(모임장) 탈퇴 시 모임 전체 + B의 편지까지 통째로 삭제됨 (B는 아무 행동 안 했는데 데이터 손실)
  - **수정**: Edge Function에 소유권 자동 이전 로직 추가 — 탈퇴하는 사용자가 방장인 모임에 다른 멤버가 있으면, 가장 먼저 들어온 멤버에게 `owner_id`/`role` 자동 이전 후 계정 삭제 진행. 멤버가 자기 혼자뿐이면 기존대로 모임째 삭제
  - **재배포 후 재테스트로 확인**: 모임 생존, B 소유권 승계, B의 편지 생존, A의 편지만 삭제(본인 것이라 정상), A 재로그인 실패(완전삭제 확인), B 셀프삭제(단독 소유자 케이스)도 정상 동작
- 테스트 스크립트는 scratchpad 임시 폴더에만 있고 저장소에는 없음(재현 필요시 새로 작성)
- Supabase 배포에 쓴 개인 액세스 토큰은 임시 파일에 잠깐 저장 후 즉시 삭제 — **사용자가 Supabase 대시보드에서 해당 토큰 revoke 권장**

### ⚠️ 계정 삭제 기능 — 나중에 실기기 테스트할 때 함께 확인할 것
API 레벨 테스트는 통과했지만, 아래는 확률은 낮아도 실기기/실사용 흐름에서 한 번 더 봐두면 좋은 것들:
1. **탈퇴 도중 네트워크가 끊기는 경우**: `delete-account` 함수 안에서 기도기록/프로필까지는 지워졌는데 마지막 `auth.admin.deleteUser()` 호출만 실패하면, 로그인은 되는데 프로필이 없는 어중간한 상태가 될 수 있음 → 이 경우 앱이 신규가입 화면으로 보내는지, 이상하게 멈추진 않는지 실기기에서 비행기 모드 전환 등으로 재현해볼 것
2. **모임 소유권 자동 승계 시 알림 없음**: 방장이 탈퇴하면 가장 먼저 들어온 멤버에게 동의 없이 방장 권한이 넘어감(데이터 보존이 우선이라 의도적으로 이렇게 처리) — 실기기에서 그 멤버 계정으로 로그인해 그룹 상세 화면이 "내가 방장"으로 정상 표시되는지, UI가 깨지지 않는지 확인. 필요하면 승계 알림 기능은 이후 세션에서 별도 추가 고려
3. **Flutter Web 빌드는 미대응**: 이 Edge Function엔 CORS 헤더가 없어서 브라우저(Flutter Web)에서 직접 호출하면 막힘 — 지금은 Android/Play스토어만 타겟이라 무관하지만, 나중에 웹 빌드를 추가하게 되면 이 함수부터 다시 손봐야 함

사용자가 `docs/다국어_지원_상세보고서.md`(claude.ai 대화 기반, 실제 코드 미확인 상태로 작성된 문서)를 근거로 "영어권 사용자에게 노출도 되고 앱 실행 시 처음부터 영어로 보이게" 작업을 요청. 실제 코드/git 이력을 확인한 결과 그 보고서의 전제("레이어2 앱 내부 다국어 미구현")가 **틀린 것으로 확인됨** — 착오 소지가 있어 다음 세션 참고용으로 정정 기록.

### 확인 결과 — 이미 완료돼 있던 것
- **앱 전체 영어화(레이어2)는 2026-07-04에 이미 완료** (`51d9dfa` ~ `d55547f` 등 `feat(i18n):` 커밋 8개, 이후에도 `71b73ad`~`51cffe8`까지 settings_kit/폰트피커/피드백/프로필수정/알림설정/알림시간피커까지 전부 완료). `CHANGELOG.md` 2026-07-04 항목에 완료 기록 있음 (`flutter analyze` 통과 + `[가-힣]` grep으로 UI 문자열 내 한글 잔존 0건 검증 완료).
- 설정 화면에서 언어(한국어/English) 수동 전환 가능 (`languageProvider`, `app_settings_screen.dart`).
- 영어 스토어 등록 문구 초안 `docs/store_listing_en.md` — 이미 다듬어진 버전으로 존재 (앱 이름/짧은 설명 2안/자세한 설명 전부 준비됨).
- 영어 개인정보처리방침(`privacy_policy_en.html`) / 계정삭제 안내(`account_deletion_en.html`) — 이미 GitHub Pages에 호스팅됨.
- `다국어_지원_상세보고서.md`는 이 사실들을 반영하지 못한 **오래되고 부정확한 문서** — 삭제하지는 않았으나 향후 참고 시 주의 필요(코드 우선 확인할 것).

### 진짜 남아있던 문제와 오늘 수정한 것
- **원인**: `lib/providers/settings_provider.dart`의 `LanguageNotifier`가 기본값을 `AppLanguage.ko`로 하드코딩 — 기기 시스템 언어와 무관하게 항상 한국어로 시작, 사용자가 설정에서 수동으로 바꿔야만 영어로 전환됨. 즉 "노출은 레이어1(스토어)만 하면 되지만, 실행 시 자동으로 영어가 나오게 하려면" 이 부분이 빠져있던 것.
- **수정 (커밋 전, 작업 트리에 반영됨)**: `_detectDeviceLanguage()` 신설 — 최초 실행 시 `PlatformDispatcher.instance.locale.languageCode`가 `'ko'`가 아니면 `AppLanguage.en`으로 자동 폴백. 단, 사용자가 설정에서 언어를 명시적으로 선택해 `SharedPreferences`에 저장한 적이 있으면 그 값이 항상 우선(기존 `_load()` 로직 그대로 유지, orElse 폴백만 하드코딩된 `ko` 대신 감지된 `state` 사용하도록 수정).
- 코드 내 오래된 주석("현재는 선택값 저장만 — 실제 다국어 번역은 추후 작업")도 실제 상태에 맞게 정정.
- `flutter analyze` 재실행 확인 — 기존과 동일한 23건(전부 무관한 `withOpacity` deprecation info)만 있고 신규 이슈 없음.
- **아직 커밋 안 함** — 사용자가 release 빌드 테스트/AAB 재빌드를 "나중에 합쳐서" 하자고 해서, 커밋 시점도 그때 맞춰 진행 예정.
